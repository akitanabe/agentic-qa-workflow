# マネージャー + QA としての委譲

委譲は速いが、丸投げは品質を壊す。worker は局所最適なコードを書きがちで、特にテストは
「正常系だけ」「実装をなぞるだけ」「comment が実態とズレる」欠陥を持ったまま返ってくる。

このスキルの前提: **生成物の品質責任は委譲先ではなく親（あなた）にある。** worker は「手」であり、
設計・タスク分割・受け入れ判断・最終検証は親 Codex エージェントが握る。「できました」は報告であって
受け入れ基準ではない。役割を一文で言うと **親 = マネージャー（分割・指示）＋ QA プランナー
（受け入れ基準の設計と検証）／worker = 実装者。**

この Codex plugin は Claude Code の `Agent` や `subagent_type` 名に依存しない。現在の Codex 環境で
サブエージェントや multi-agent tool が使える場合は、その tool contract に従って使う。使えない場合は
委譲したふりをせず、親が直接作業するか、要求された委譲面が利用できないことを報告する。

## Worker 確認

最初に、委譲計画や全体フローへ進む前に Codex の custom agent リストを確認する。リストに
`implementer`、`senior-implementer`、`responsibility-boundary-reviewer`、`refactor-patch-agent` が
すべて表示されるなら、この確認だけで次へ進む。余分な filesystem 確認やコピーはしない。

Codex の custom agent 定義は plugin 有効化だけでは自動登録されない。agentic-qa-workflow plugin は
`install/agents/*.toml` に custom agent 定義を同梱するので、リストに不足がある場合だけ plugin の
install 先から、plugin の利用スコープに合う custom agent directory へコピーする。

`PLUGIN_DIR` は、source checkout の `plugins/codex` または Codex が plugin を install した directory
に置き換える。`AGENT_DIR` は、ユーザーインストールとして使うなら `~/.codex/agents`、repo に閉じて
使うなら `<repo>/.codex/agents` にする。

```text
PLUGIN_DIR=<agentic-qa-workflow plugin directory>
AGENT_DIR=~/.codex/agents
# or: AGENT_DIR=<repo>/.codex/agents
mkdir -p "$AGENT_DIR"
cp "$PLUGIN_DIR"/install/agents/*.toml "$AGENT_DIR"/
```

コピーを実行した場合は、custom agent を現在の session へ確実に反映するため、ユーザーに Codex session
の再起動を依頼し、その時点で処理を打ち切る。再起動後の新しい session で改めて依頼を受けてから、
委譲計画と実装へ進む。

## 全体フロー

1. **分割する** — タスクを「共有の土台」と「直列に受け入れる枝」に分ける。
2. **土台を作り、検証してコミットする** — 共有基盤（テスト基盤・fixture・テストデータ・定数）は
   親が先に1つ確定させ、動作確認して**最初の baseline コミット**にする。土台自体を前段タスクとして扱う。
3. **枝をキュー化する** — 依存関係・リスク・検証しやすさで順番を決める。前の枝の判断や修正を次の枝に
   反映できる順番にする。
4. **先頭の枝だけ委譲する** — 最新の baseline から、可能なら隔離された worktree を用意し、枝ごとに難度を
   見積もって worker を選ぶ。「worktree による枝の隔離」を参照。
5. **返却物を精査する（QA）** — 変更 diff とテストを実際に読み、品質ルーブリックで欠陥を洗い出す。
   精査の深さは枝の種別に応じて変える（「枝の種別ごとの TDD 強度」を参照）。
6. **責務境界レビューゲートを通す** — 返却物 QA の後、責務混在・境界違反・副作用分散を軽量チェックし、
   条件に該当する差分は専用 reviewer に渡す（「責務境界ゲート」を参照）。
7. **取り込み、自分の手で検証する** — 統合用ブランチへ1枝だけ取り込む。具体手順は
   「枝の返却と統合メカニクス」に従う。
8. **baseline を前進させて次の枝へ進む** — 統合後に green になったコミットを次の枝の baseline にする。
9. **`/review` を実行する** — サイクル最後に、環境が提供するなら現在の `/review` workflow を実行する。
   なければ同等の統合済み diff review を親が行い、その結果を最終報告に含める。

### baseline はコミットで進める

作業は **枝を1つ受け入れるたびに baseline を前進させる** 繰り返しで進める。次の枝は常に更新後の
baseline コミットから始め、未コミット変更を渡す前提にしない。差し戻す枝があっても、取り込み済みの枝を
巻き込まない。

baseline が常にコミット済みなら、隔離した worktree は必ず確定済みの状態から作られ、未コミット変更が
紛れ込む余地がない。統合と検証の具体手順は「枝の返却と統合メカニクス」に寄せ、タスク完了時に必要なら
レビュー・revert しやすいコミット粒度へ整える。

### 枝の種別ごとの TDD 強度

TDD/QA のセレモニーにはコストがある。全枝に段階ゲートを課せば往復が増えて軽い枝では割に合わず、
逆に代償の大きい枝を丸投げすれば穴が残る。だから一律にはしない。**枝の種別を見極め、担保の強度を
そこに合わせる。** 判断軸は行数ではなく「誤実装の代償」と「仕様の明確さ」。

| 枝の種別 | TDD 強度 |
| --- | --- |
| **軽い修正・明確な仕様** | worker を信頼。親は返却の diff とテストだけ確認して締める。段階ゲートも AC 対応表も課さない。 |
| **通常実装（既定）** | 委譲プロンプトで AC→テスト対応表・境界値・異常系・Red 時点の失敗出力を要求し、返却を QA ルーブリックで精査、親が green を確認して担保する。 |
| **重要・高リスク実装** | 一気通貫で投げず段階ゲート（テスト計画→失敗テスト→実装）に割る。下記手順。 |
| **仕様が曖昧** | 実装させる前に、仕様確認とテスト計画だけ先に返させて親が見る。曖昧なまま実装に入らせない。 |

「軽い」に振ってよいのは、仕様が明確で影響範囲が閉じ、誤ってもすぐ気づいて戻せる枝に限る。迷ったら
一段重い側に倒す（取りこぼしを親が手直しするコストの方が高くつく）。**既存テストに触れる枝・共有基盤に
関わる枝は「軽い」に入れない** — 波及範囲が読みにくく、壊れたときの代償が局所に収まらないため。

**重要・高リスク実装の段階ゲート。** 最終成果物だけ受け取る運用では Red→Green→Refactor を踏んだか
検証できず、worker の能力に関わらず「TDD でやりました」と報告するだけになりやすい（`senior-implementer`
相当に投げても、最終 diff だけでは Red を先に書いたか親には確かめられない）。だから段階に割る：

1. **テスト計画だけ返させる** — どの境界・異常系をどう検証するかの一覧。親が承認。
2. **失敗するテストだけ書かせる** — 実装はまだ。親が「狙いどおり fail する」ことを自分で確認。
3. **実装させる** — テストを通す。後からテストを実装に合わせ書き換えていないか親が確認。

段階ゲートを使う枝は、継続できる環境では**同じ worker（同じ worktree）を継続**して進める。新しい worker
を起動すると別 worktree になり、実装フェーズが失敗テストを見ないまま始まる。継続できない場合は、
親が失敗テストをその枝に取り込んでコミットしてから実装を委譲する。

### worktree による枝の隔離

利用できる場合は、委譲する枝ごとに専用 git worktree を使い、隔離された worktree で worker に委譲する。
worktree の目的は並列速度ではなく、枝ごとの作業状態を親が読みやすい diff として受け取り、問題があれば
枝ごと差し戻せるようにすること。

- **次の枝だけを最新の baseline から作る** — 未コミットの土台変更が枝に伝播する前提にしない。
- **共有変更は先に土台へ寄せる** — 共有 fixture・設定・lockfile・生成物・共通 helper などを複数の枝が
  必要とするなら、親が前段タスクとして確定してから枝を切る。複数 worker に同じ土台を別々に再実装させない。
- **1枝の責務を小さく保つ** — 同じ枝に複数の変更理由を詰め込むと QA と差し戻しが難しくなる。
- **外部共有リソースは worktree では隔離されない** — DB・Redis・queue・port・共有 temp・生成 cache・
  `.env`・外部 API mock などを使う枝は、専用リソースを割り当てるか、親が検証時に衝突しない状態へ整理する。
- **取り込みは1枝ずつ、順番に** — 返却・QA・cherry-pick・統合後検証は「枝の返却と統合メカニクス」に従う。

worker には次を返させる。

- worktree path と branch 名
- baseline commit
- 返却 commit SHA range
- 変更ファイル
- 実行した command と結果要約
- 未コミット変更が残る場合は、その内容と理由

### 枝の返却と統合メカニクス

直列受け入れは **コミット単位で返し、親が diff を読んでから cherry-pick する** 運用に固定する。
作業ツリーに置いたまま返すと、親が対象 diff と統合方法を即興で補うことになり、枝単位の差し戻しも
曖昧になるため。

1. **委譲直前に baseline の green を確認し、親が baseline を渡す** — まず現在の baseline コミットで既存の
   検証 command（既存テスト・build・typecheck・lint 等）が green であることを親が確認する（baseline が
   赤いままだと、worker が自分由来でない失敗の調査に浪費し、返却の信頼性も判定できない）。枝が新規に
   追加するテストへの focused test は baseline では実行できないため、返却前（worker）と統合後（親）の
   検証に充てる。そのうえで baseline コミット SHA、枝の種別、worktree 内で作業していること、返却時に
   コミットすることをプロンプトへ明記する。隔離された worktree の実際のパスとブランチ名は起動後に
   決まるため、worker 自身に確認して返却物へ含めさせる。
2. **worker は枝の worktree でコミットして返す** — 検証済みの変更を1つ以上のコミットにし、worktree パス、
   作業ブランチ名、先頭/末尾コミット SHA、実行した検証を返却物に含める。未コミット変更が残る場合は
   未完了として理由を明記する。
3. **親は枝の worktree で diff を読む** — `git -C <worktree> status --short` で未コミット変更の有無を見て、
   `git -C <worktree> diff <base>...HEAD` と対象テストを開いて QA する。報告だけで受け入れない。
4. **責務境界 reviewer には diff テキストを渡す** — 新しい worker は別 worktree で始まり枝の変更を見ない。
   `responsibility-boundary-reviewer` には対象コミット範囲、変更ファイル一覧、`git diff <base>...HEAD` の
   内容に加え、その枝のタスク要約と AC も渡す（要件由来の変更を責務混在と誤判定させないため）。reviewer に
   作業ツリーの存在を前提にさせない。
5. **差し戻しは同じ枝で継続する** — QA hard reject や reviewer 指摘で修正が必要なら、原則として同じ worker
   conversation を継続して差し戻し、同じ worktree に修正コミットを追加させる。スコープ逸脱が大きい、
   baseline コミットを誤った、継続不能など枝を救う方が危ない場合だけ、その枝を捨てて現在の baseline
   コミットから新規 worktree でやり直す。統合ブランチ上で未受け入れ枝を直接直さない。
6. **refactor patch worker は継続不能時の代替にする** — 元の実装 worker を継続できない、または修正が
   reviewer の指摘範囲だけに閉じた最小リファクタである場合に限り、`refactor-patch-agent` を使う。
   その場合は枝の worktree パスと対象コミットを明記し、その worktree 上で最小修正コミットを追加させる。
7. **親は統合用ブランチへ1枝だけ取り込む** — QA と責務境界ゲートを通過したら、統合用ブランチで
   `git cherry-pick <first>^..<last>`（単一コミットなら `git cherry-pick <sha>`）を使い、1枝だけ取り込む。
   merge commit は作らない。コンフリクトしたら即興で広く直さず、baseline 差分を確認して表層以外は差し戻す。
8. **統合後に親が再検証する** — 枝 worktree で green でも、統合ブランチ上で focused test と関連する
   build/typecheck/lint を実行する。ここで落ちた場合、表層の import・comment・format だけ親が直し、
   境界値漏れ・仕様ズレ・既存テスト破壊・設計判断を伴う修正は差し戻す。
9. **green を baseline コミットにする** — 統合後の検証が通った状態をチェックポイントとしてコミット済みにし、
   そのコミットを次の枝の baseline にする。未コミット変更を次の枝へ暗黙に渡さない。

枝 worktree と統合ブランチの検証は目的が違う。枝 worktree の検証は「その枝の返却物が自己完結しているか」
を見るため、統合ブランチの検証は「受け入れ済み baseline と組み合わせても壊れないか」を見るために行う。
片方が green でももう片方を省略しない。

## 難度に応じた worker 選択

委譲先は難度で使い分ける。難度は「行数」でなく**判断の難しさ**で測る — 設計の自由度・影響範囲・
ドメインの罠が多いほど上。custom agents が登録済みなら次の agent 名を優先して使う。登録されていない、
または現在の Codex 環境で agent 名を指定できない場合は、plugin 固有の agent 名に依存せず、prompt 内で
同等の role profile を使い分ける。

| custom agent / role profile | 使う場面 |
| --- | --- |
| `implementer` / 通常 implementer | 仕様が明確で範囲が閉じた通常の実装・テスト追加（大半の枝）。 |
| `senior-implementer` / senior implementer | 設計判断を伴う／複数 module に波及する／非自明な algorithm・concurrency／誤実装の代償が大きい枝。 |
| `responsibility-boundary-reviewer` / responsibility-boundary reviewer | 実装済み diff の責務混在・境界違反・副作用分散を確認する。ファイルは編集しない。 |
| `refactor-patch-agent` / refactor patch worker | responsibility-boundary reviewer の指摘を解消する最小 patch だけを作る。 |

- **迷ったら senior 側。** 取りこぼしを親が手直しするコストの方が高くつく。難所と雑務が混在するなら
  枝を割る（難所は senior implementer／雑務は通常 implementer）。
- **既存テストに触れる枝・共有基盤に関わる枝は通常 implementer に入れない** — 波及範囲が読みにくく、
  壊れたときの代償が局所に収まらない。
- **親が列挙する境界値・異常系はどちらに渡すかで変えない。** 同じ密度で書く。

同じ worker conversation を継続できる環境では、段階ゲートや差し戻しにそれを使う。継続できない場合は、
親が受け入れ済みの失敗テストや review input を枝へコミットしてから次フェーズを委譲する。

## 委譲プロンプトの組み立て

worker は親の文脈を持たない。下記テンプレートをタスクに合わせて埋めて渡す。

```markdown
## 役割
あなたはこの枝の <通常 implementer / senior implementer> です。下記タスクのコードとテストを書いて
ください。設計判断で迷ったら、勝手に仕様を広げず、前提と判断点を明記して返すこと。

## タスク
<実装する振る舞いを1〜数文で。内部構造ではなく「外から見て何ができるか」で書く>

## 実行コンテキスト
- 枝の種別: <軽い修正・明確な仕様 / 通常実装 / 重要・高リスク実装 / 仕様が曖昧>
- baseline commit: <SHA>
- 検証 command: <focused test / build / typecheck / lint の具体 command。親が package.json・Makefile・
  CI 設定などの正から確認して必ず渡す>
- 作業環境: 隔離された worktree 内で作業している。実際の worktree パスと作業ブランチは起動後に自分で
  確認し、返却物に含めること。
- 完了時: 検証済みの変更を commit し、baseline commit・返却 commit SHA range も返すこと。

## 受け入れ条件（AC）
- AC-1: <条件>
- AC-2: <条件>
通常実装・重要実装では、返却時に「AC-n → それを検証する test 名 → 期待値の根拠（仕様のどこから
導いたか）」の対応表を必ず付けること。軽い修正では、親が明示した場合だけ対応表を付けること。

## 対象とスコープ
- 触ってよい file: <paths>
- 触ってはいけない file: <paths。共有土台は親が管理>
- 再利用する共有の土台: <fixtures/helpers/test data の場所。再実装せず使う>
- 禁止（守れない返却は不可）: 既存 test の削除・skip・期待値変更をしない／新規依存を追加しない／
  触る理由のない file を変更しない（いずれも必要なら判断点として返す）

## テストの品質基準（満たさない返却は不可）
- 振る舞いベース: 外部から観測可能な振る舞いを test する。内部実装（private 直叩き・実装手順の
  なぞり）に依存せず、仕様変更に強くする。
- 網羅性: 正常系だけでなく境界値・異常系・分岐網羅を含める。
  最低限 cover するケース: <null/empty/zero/max/min/invalid/exception paths を具体列挙>
- TDD: Red→Green→Refactor。失敗する test を先に書き、最小実装で通し、後付けで実装に合わせない。
  Red を確認した時点のテスト失敗出力を控えて返却物に含める。可能なら Red と Green で commit を分ける。
- comment: 自明でないロジック・制約・前提・test の意図を残す。コードと矛盾する comment は書かない。

## 返却物
- worktree パス、作業ブランチ、baseline commit、返却 commit SHA range（範囲があれば先頭/末尾）
- 変更/追加 file 一覧
- 「AC → test 名」「test → 見ている境界/異常系」「期待値の根拠（仕様のどこから導いたか）」の
  対応表（軽い枝では親が求めた場合のみ）
- 実行した検証 command（test/build/typecheck/lint 等）と結果の要点。返却前に focused test を必ず
  実行し、実行できない場合は理由を添えて未完了として返すこと
- Red を確認した時点のテスト失敗出力（抜粋。軽い枝では親が求めた場合のみ）
- 前提・判断点・cover しきれなかったケース

## 出力先
<保存先パス。あれば>
```

**最も価値があるのは境界値・異常系を親が列挙すること。** ここを丸投げすると抜ける。親がドメインを
理解して「null のとき」「上限超過」「0 件」「例外が飛ぶ経路」を具体的に書き出す。共有基盤の場所も
必ず渡す（渡さないと worker が再実装する）。

## 返却物の QA ルーブリック（親が必ず手を動かす）

報告を信じず、**変更 diff とテストを自分で開いて読む**。委譲で品質が落ちる典型はテスト不足だけで
なく、実装の scope creep・既存設計違反・expected value の捏造・既存テスト弱体化も同じくらい危ない。

以下は**通常実装・高リスク実装で全項目を手を動かして確認する**基準。**軽い修正・明確な仕様**の枝は
観点0（diff を読む）と観点5（自分で green を確認）に絞ってよい — 境界値の網羅列挙まで課すと割に
合わないため。ただし「軽い」と判断した前提が崩れたら（想定外の file に手が入っている等）、その場で
通常の全項目に引き上げる。

**0. 実装そのもの（diff を読む）**
- scope creep（依頼外の file/振る舞い）／既存パターン（設計・命名・error handling）からの逸脱
- 公開契約（API・signature・戻り値）の破壊／既存 test の削除・skip・期待値変更による「通し」
- 依存の無断追加／非機能の罠（例外処理・resource 解放・concurrency・security）

**1. 振る舞いベースか** — 内部構造に密結合し、実装を1行変えただけで無関係に落ちる脆さがないか。

**2. 網羅性** — 正常系だけは未完成。親が列挙した境界/異常ケースが**実際に**テストになっているか
（名前と中身を突き合わせる）。expected value・オフセット・比較対象は、返却物の「期待値の根拠」列と
仕様を突き合わせて確かめる（「通る expected value」を逆算した捏造に注意）。

**3. TDD の健全性** — 実装内部を不自然に写した assert がないか。リファクタ後も意味を保つか。
返却物の Red 時点の失敗出力（または Red/Green に分かれた commit 履歴）で、テストが先に落ちたことを
確認する。

**4. コメントの正しさ** — コードと一致しているか。誤った機構説明・誤 comment は親が直す。

**5. 自分の手で実行** — 検証 command（test/build/typecheck/lint 等）を親が実行し green を確認。既存
test を壊していないか（独立・全体の両方）。

## 責務境界ゲート

返却物 QA を通過した diff は、最終検証に進む前に親が責務境界の軽量チェックを行う。通常 QA に埋もれ
やすい設計上の問題をここで分離して扱う。軽い修正では diff を読む中で明らかな責務逸脱がないかだけを
確認すればよく、下記7項目すべてを個別に証明させる必要はない。通常実装以上、または起動条件に該当する
差分では `responsibility-boundary-reviewer` を使う。

### 親が常に見る軽量チェック

- 1つの関数・class・module に複数の変更理由が混ざっていないか。
- 入力検証、業務判断、永続化、外部 I/O、表示・レスポンス整形が同じ場所に詰め込まれていないか。
- DB、API、HTTP、file、framework などの具体実装を上位層が知りすぎていないか。
- 副作用が分散し、再実行・テスト・失敗時の扱いが難しくなっていないか。
- boolean flag や mode argument、巨大な条件分岐で責務を切り替えていないか。
- 既存の責務配置・命名・directory 構成から不自然に外れていないか。
- 分割や抽象化が、責務分離ではなく過剰抽象化になっていないか。

### 専用 reviewer の起動条件

次のいずれかに該当する場合は `responsibility-boundary-reviewer` に実装済み diff を渡す。

- 複数層にまたがる変更。
- DB、API、HTTP、file、mail、time、random、queue などの外部 I/O や永続化の追加。
- boolean flag や mode argument で大きく処理を切り替える変更。
- 新しい抽象化、layer、adapter、service、manager、handler、processor などの追加。
- 1つの差分に入力整理、業務判断、副作用、出力整形が混在している疑いがある変更。
- 親 QA で責務混在・境界違反・副作用分散が疑わしいが、通常の correctness check だけでは判断しにくい変更。

reviewer には commit range、変更 file list、`git diff <base>...HEAD`、task summary、acceptance criteria
だけを渡す。reviewer が枝 worktree を見られる前提にしない。reviewer の出力は次の形式にする。

1. 全体判定: `問題なし`、`軽微`、`修正推奨`、`修正必須`
2. 指摘: location、type、reason、impact、minimal fix
3. 過剰抽象化を避ける注意点
4. diff 外の既存課題。判定とは分ける

### 判定の扱い

- `問題なし`: ゲート通過。親は通常の最終検証へ進める。
- `軽微`: 親が残課題として明示するか、表層修正で収まる場合だけ親が直す。
- `修正推奨`: 原則として同じ worker conversation を継続して差し戻す。継続不能、または reviewer の
  指摘範囲だけに閉じた最小リファクタなら refactor patch worker に渡す。
- `修正必須`: 同じ枝で解消するまで最終完了にしない。継続不能、または reviewer の指摘範囲だけに閉じた
  最小リファクタなら refactor patch worker に渡す。

### agent の責務分離

`responsibility-boundary-reviewer` は修正しない。全体判定と、指摘ごとの（判定・問題箇所・種類・理由・
影響範囲・最小修正方針）、やりすぎな抽象化を避ける注意点だけを返す（全体判定は最も重い指摘に合わせる）。
指摘は diff が導入・悪化させた問題に限り、diff に含まれない既存コード由来の問題は「既存課題」として
判定と区別して報告させる。

`refactor-patch-agent` は reviewer の指摘範囲だけを最小修正する。仕様変更、ついで修正、大規模再設計、
新規依存追加、通常実装の代行はしない。修正後は実行した検証と残課題を返す。

## 受け入れ・差し戻しの判断

**親がその場で直してよいのは判断を伴わない表層だけ** — import 整理・命名・comment・format。ここは
往復より親が直す方が速く、品質責任は親にある原則とも合う。

**次は hard reject（直さず差し戻す）。** 親がこっそり直すと穴が記録されず再発する：
境界値・異常系 test の欠落／expected value の根拠不明・捏造の疑い／禁止 file 変更・既存 test 弱体化・
未承認 dependency／依頼にない仕様拡張・scope creep／test 未実行・失敗 test 放置。
差し戻すときは「何が・なぜ受け入れ不可で・どう直すか」を AC/観点に紐づけて具体的に書く（「もっと網羅
して」は同じ欠陥を生む）。

差し戻しは原則として同じ worker conversation を継続して返し、同じ worktree に修正 commit を追加させる。
枝の前提自体が壊れている場合だけ、その枝を捨てて現在の baseline コミットから新規 worktree でやり直す。

## 最終報告

タスク完了時の報告には次を含める。

- 変更内容
- worker が検証したこと
- 統合後に親が検証したこと
- `/review` または同等の統合済み diff review で確認したこと
- 未検証の残り

## 補足

言語・framework 非依存。プロジェクト固有の検証 command（test/build/typecheck/lint 等）や検証ルールが
ある場合は、委譲プロンプトの「テストの品質基準」と QA の「自分の手で実行」にその具体を差し込んで使う。
