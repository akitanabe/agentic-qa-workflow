# マネージャー＋QA としての委譲

委譲は速いが、丸投げは品質を壊す。サブエージェントは局所最適なコードを書きがちで、特にテストは
「正常系だけ」「実装をなぞるだけ」「コメントが実態とズレる」欠陥を持ったまま返ってくる。

このスキルの前提：**生成物の品質責任は委譲先ではなく親（あなた）にある。**
サブエージェントは「手」、設計・分割・受け入れ判断・最終検証は親が握る。「できました」は報告で
あって受け入れ基準ではない。役割を一文で言うと **親 = マネージャー（分割・指示）＋ QA プランナー
（受け入れ基準の設計と検証）／サブエージェント = 実装者。**

## 全体の流れ

1. **分割する** — タスクを「共有の土台」と「直列に受け入れる枝」に分ける。
2. **土台を作り、検証してコミットする** — 共有基盤（テスト基盤・フィクスチャ・テストデータ・定数）は
   親が先に1つ確定させ、動作確認して**最初の基準コミット**にする。土台自体を前段タスクとして扱う。
3. **枝をキュー化する** — 依存関係・リスク・検証しやすさで順番を決める。前の枝の判断や修正を次の枝に
   反映できる順番にする。
4. **先頭の枝だけ worktree で委譲する** — 最新の基準コミットから専用 worktree を作り、枝ごとに難度を
   見積もって implementer を選ぶ。「worktree による枝の隔離」を参照。
5. **返却物を精査する（QA）** — 変更 diff とテストを実際に読み、品質ルーブリックで欠陥を洗い出す。
   精査の深さは枝の種別に応じて変える（「TDD 担保は枝の種別で強度を変える」を参照）。
6. **責務境界レビューゲートを通す** — 返却物 QA の後、責務混在・境界違反・副作用分散を軽量チェックし、
   条件に該当する差分は専用 reviewer に渡す（「責務境界レビューゲート」を参照）。
7. **取り込み、自分の手で検証する** — 統合用ブランチへ1枝だけ取り込み、検証コマンド
   （test/build/typecheck/lint 等）を親が実行して green を確認。表層なら直し、それ以外は差し戻す。
8. **基準を前進させて次の枝へ進む** — green の枝をチェックポイントとしてコミットし、残った曖昧さ・
   誤コメント・抜け漏れを親が直してから次の枝を始める。

### 基準はコミットで進める

作業は **枝を1つ受け入れるたびに基準を前進させる** 繰り返しで進める：

1. **現在の基準コミットから次の枝の worktree を作る** — 未コミット変更を渡す前提にしない。
2. **その枝だけを委譲し、QA して取り込む** — green になった枝はチェックポイントとしてコミットする。
   差し戻す枝があっても取り込み済みの枝を巻き込まない。
3. **次の枝は更新後の基準コミットから始める** — 前の枝で確定した仕様判断・テスト基盤・修正を反映した
   状態を次の作業開始点にする。
4. **タスク完了時にコミット粒度を整える** — 必要ならチェックポイントを squash して、レビュー・revert
   しやすいタスク単位のコミットにする。

基準が常にコミット済みなら、worktree は必ず確定済みの状態から作られ、未コミット変更が紛れ込む余地が
ない。枝単位のチェックポイントは差し戻しに備えるためのものなので、最終のコミット粒度はタスクの大きさに
合わせて整える。

### TDD 担保は枝の種別で強度を変える

TDD/QA のセレモニーにはコストがある。全枝に段階ゲートを課せば往復が増えて軽い枝では割に合わず、
逆に代償の大きい枝を丸投げすれば穴が残る。だから一律にはしない。**枝の種別を見極め、担保の強度を
そこに合わせる。** 判断軸は行数ではなく「誤実装の代償」と「仕様の明確さ」。

| 枝の種別 | TDD 担保の強度 |
|----------|----------------|
| **軽い修正・明確な仕様** | エージェントを信頼。親は返却の diff とテストだけ確認して締める。段階ゲートも AC 対応表も課さない。 |
| **通常実装（既定）** | 委譲プロンプトで AC→テスト対応表・境界値・異常系を要求し、返却を QA ルーブリックで精査、親が green を確認して担保する。 |
| **重要・高リスク実装** | 一気通貫で投げず段階ゲート（テスト計画→失敗テスト→実装）に割る。下記手順。 |
| **仕様が曖昧** | 実装させる前に、仕様確認とテスト計画だけ先に返させて親が見る。曖昧なまま実装に入らせない。 |

「軽い」に振ってよいのは、仕様が明確で影響範囲が閉じ、誤ってもすぐ気づいて戻せる枝に限る。迷ったら
一段重い側に倒す（取りこぼしを親が手直しするコストの方が高くつく）。**既存テストに触れる枝・共有基盤に
関わる枝は「軽い」に入れない** — 波及範囲が読みにくく、壊れたときの代償が局所に収まらないため。

**重要・高リスク実装の段階ゲート。** 最終成果物だけ受け取る運用では Red→Green→Refactor を踏んだか
検証できず、implementer の能力に関わらず「TDD でやりました」と報告するだけになりやすい（`senior-implementer`
に投げても、最終 diff だけでは Red を先に書いたか親には確かめられない）。だから段階に割る：

1. **テスト計画だけ返させる** — どの境界・異常系をどう検証するかの一覧。親が承認。
2. **失敗するテストだけ書かせる** — 実装はまだ。親が「狙いどおり fail する」ことを自分で確認。
3. **実装させる** — テストを通す。後からテストを実装に合わせ書き換えていないか親が確認。

段階ゲートを使う枝は**同じサブエージェント（同じ worktree）を `SendMessage` で継続**して進める。
新規の `Agent` 呼び出しは別 worktree になり、実装フェーズが失敗テストを見ないまま始まる。継続しない
場合は、親が失敗テストをその枝に取り込んでコミットしてから実装を委譲する。

### worktree による枝の隔離

各枝は専用の git worktree で動かす。`Agent` ツールの `isolation: "worktree"` を指定すると別ブランチの
隔離コピーで作業する（未変更なら自動で片付く）。worktree の目的は速度ではなく、枝ごとの作業状態を
親が読みやすい diff として受け取り、問題があれば枝ごと差し戻せるようにすること。

- **次の枝だけを最新の基準コミットから作る** — 未コミットの土台変更が枝に伝播する前提にしない。
- **共有変更は先に土台へ寄せる** — 共有 fixture・設定・ロックファイル・自動生成物・共通 helper などを
  複数の枝が必要とするなら、親が前段タスクとして確定してから枝を切る。
- **1枝の責務を小さく保つ** — 同じ枝に複数の変更理由を詰め込むと QA と差し戻しが難しくなる。
- **外部共有リソースは worktree では隔離されない** — DB・Redis・キュー・ポート・共有 temp・生成キャッシュ・
  `.env`・外部 API モックなどを使う枝は、専用リソースを割り当てるか、親が検証時に衝突しない状態へ整理する。
- **取り込みは1枝ずつ、順番に**（「枝の返却と統合メカニクス」参照）— 取り込むたびに focused test と関連
  する検証（build/typecheck/lint 含む）で green を確認。コンフリクトや既存テスト破壊が出たら差し戻す。

### 枝の返却と統合メカニクス

直列受け入れは **コミット単位で返し、親が diff を読んでから cherry-pick する** 運用に固定する。
作業ツリーに置いたまま返すと、親が対象 diff と統合方法を即興で補うことになり、枝単位の差し戻しも
曖昧になるため。

1. **委譲時に親が基準を渡す** — 基準コミット SHA、想定ブランチ名、worktree 内で作業していること、
   返却時にコミットすることをプロンプトへ明記する。
2. **サブエージェントは枝の worktree でコミットして返す** — 検証済みの変更を1つ以上のコミットにし、
   worktree パス、作業ブランチ名、先頭/末尾コミット SHA、実行した検証を返却物に含める。未コミット変更が
   残る場合は未完了として理由を明記する。
3. **親は枝の worktree で diff を読む** — `git -C <worktree> status --short` で未コミット変更の有無を見て、
   `git -C <worktree> diff <base>...HEAD` と対象テストを開いて QA する。報告だけで受け入れない。
4. **責務境界 reviewer には diff テキストを渡す** — 新規 `Agent` は別 worktree で始まり枝の変更を見ない。
   `responsibility-boundary-reviewer` には対象コミット範囲、変更ファイル一覧、`git diff <base>...HEAD` の内容を
   プロンプトに貼る。reviewer に作業ツリーの存在を前提にさせない。
5. **refactor patch は同じ枝で継続する** — 指摘修正が必要なら、原則として同じサブエージェントへ `SendMessage`
   で差し戻す。新規 `refactor-patch-agent` を使う場合は、枝の worktree パスと対象コミットを明記し、その
   worktree 上で最小修正コミットを追加させる。統合ブランチ上で未受け入れ枝を直接直さない。
6. **親は統合用ブランチへ1枝だけ取り込む** — QA と責務境界ゲートを通過したら、統合用ブランチで
   `git cherry-pick <first>^..<last>`（単一コミットなら `git cherry-pick <sha>`）を使い、1枝だけ取り込む。
   merge commit は作らない。コンフリクトしたら即興で広く直さず、基準差分を確認して表層以外は差し戻す。
7. **統合後に親が再検証する** — 枝 worktree で green でも、統合ブランチ上で focused test と関連する
   build/typecheck/lint を実行する。ここで落ちた場合、表層の import・コメント・フォーマットだけ親が直し、
   境界値漏れ・仕様ズレ・既存テスト破壊・設計判断を伴う修正は差し戻す。
8. **green を基準コミットにする** — 統合後の検証が通った状態をチェックポイントとしてコミット済みにし、
   そのコミットを次の枝の基準にする。未コミット変更を次の枝へ暗黙に渡さない。

枝 worktree と統合ブランチの検証は目的が違う。枝 worktree の検証は「その枝の返却物が自己完結しているか」
を見るため、統合ブランチの検証は「受け入れ済み基準と組み合わせても壊れないか」を見るために行う。
片方が green でももう片方を省略しない。

## 難度に応じた implementer（サブエージェント）選択

委譲先は難度で2つの implementer を使い分ける。`Agent` ツールの `subagent_type` に指定して起動する。
難度は「行数」でなく**判断の難しさ**で測る — 設計の自由度・影響範囲・ドメインの罠が多いほど上。

| subagent_type | 充てる難度 |
|---------------|-----------|
| **`senior-implementer`** | 設計判断を伴う／複数モジュールに波及する／非自明なアルゴリズム・並行性／誤実装の代償が大きい枝。 |
| **`implementer`** | 仕様が明確で範囲が限定された通常の実装・テスト追加（大半の枝）。 |

- **迷ったら `senior-implementer`。** 取りこぼしを親が手直しするコストの方が高くつく。難所と雑務が混在
  するなら枝を割る（難所 `senior-implementer`／雑務 `implementer`）。
- **既存テストに触れる枝・共有基盤に関わる枝は `implementer` に入れない** — 波及範囲が読みにくく、
  壊れたときの代償が局所に収まらない。
- **親が列挙する境界値・異常系はどちらに渡すかで変えない。** 同じ密度で書く。

## 委譲プロンプトの組み立て

サブエージェントは親の文脈を持たない。下記テンプレートをタスクに合わせて埋める。

```
## 役割
あなたは実装者です。下記タスクのコードとテストを書いてください。
設計判断で迷ったら、勝手に仕様を広げず、前提と判断点を明記して返すこと。

## タスク
<実装する振る舞いを1〜数文で。内部構造ではなく「外から見て何ができるか」で書く>

## 実行コンテキスト
- 枝の種別: <軽い修正・明確な仕様 / 通常実装 / 重要・高リスク実装 / 仕様が曖昧>
- 基準コミット: <SHA>
- 作業ブランチ: <ブランチ名>
- worktree: <この Agent が作業している worktree。新規 Agent には見えない前提>
- 完了時: 検証済みの変更をコミットし、worktree パス・作業ブランチ・コミット SHA を返すこと。

## 受け入れ条件（AC）
- AC-1: <条件>
- AC-2: <条件>
通常実装・重要実装では、返却時に「AC-n → それを検証するテスト名」の対応表を必ず付けること。
軽い修正では、親が明示した場合だけ対応表を付けること。

## 対象とスコープ
- 触ってよいファイル: <パス>
- 触ってはいけないファイル: <パス。共有土台は親が管理>
- 既存の共有基盤: <フィクスチャ/ヘルパー/テストデータの場所。再実装せず使う>
- 禁止（守れない返却は不可）: 既存テストの削除・skip・期待値変更をしない／新規依存を追加しない／
  触る理由のないファイルを変更しない（いずれも必要なら判断点として返す）

## テストの品質基準（満たさない返却は不可）
- 振る舞いベース: 外部から観測可能な振る舞いをテストする。内部実装（private 直叩き・実装手順の
  なぞり）に依存せず、仕様変更に強くする。
- 網羅性: 正常系だけでなく境界値・異常系・分岐網羅を含める。
  最低限カバーすべきケース: <境界値/null/空/上限下限/例外パスを具体列挙>
- TDD: Red→Green→Refactor。失敗するテストを先に書き、最小実装で通し、後付けで実装に合わせない。
- コメント: 自明でないロジック・制約・前提・テストの意図を残す。コードと矛盾するコメントは書かない。

## 返却物
- worktree パス、作業ブランチ、基準コミット、返却コミット SHA（範囲があれば先頭/末尾）
- 変更/追加ファイル一覧
- 「AC → テスト名」「テスト → 見ている境界/異常系」の対応表（軽い枝では親が求めた場合のみ）
- 実行した検証コマンド（test/build/typecheck/lint 等）と結果の要点（未実行なら「未実行」と明記）
- 前提・判断点・カバーしきれなかったケース

## 出力先
<保存先パス。あれば>
```

**最も価値があるのは境界値・異常系を親が列挙すること。** ここを丸投げすると抜ける。親がドメインを
理解して「null のとき」「上限超過」「0 件」「例外が飛ぶ経路」を具体的に書き出す。共有基盤の場所も
必ず渡す（渡さないと再実装する）。

## 返却物の QA ルーブリック（親が必ず手を動かす）

報告を信じず、**変更 diff とテストを自分で開いて読む**。委譲で品質が落ちる典型はテスト不足だけで
なく、実装のスコープ逸脱・既存設計違反・期待値の捏造・既存テスト弱体化も同じくらい危ない。

以下は**通常実装・高リスク実装で全項目を手を動かして確認する**基準。**軽い修正・明確な仕様**の枝は
観点0（diff を読む）と観点5（自分で green を確認）に絞ってよい — 境界値の網羅列挙まで課すと割に
合わないため。ただし「軽い」と判断した前提が崩れたら（想定外のファイルに手が入っている等）、その場で
通常の全項目に引き上げる。

**0. 実装そのもの（diff を読む）**
- スコープ逸脱（依頼外のファイル/振る舞い）／既存パターン（設計・命名・エラー処理）からの逸脱
- 公開契約（API・シグネチャ・戻り値）の破壊／既存テストの削除・skip・期待値変更による「通し」
- 依存の無断追加／非機能の罠（例外処理・リソース解放・並行性・セキュリティ）

**1. 振る舞いベースか** — 内部構造に密結合し、実装を1行変えただけで無関係に落ちる脆さがないか。

**2. 網羅性** — 正常系だけは未完成。親が列挙した境界/異常ケースが**実際に**テストになっているか
（名前と中身を突き合わせる）。期待値・オフセット・比較対象の根拠を自分で確かめる（「通る期待値」を
逆算した捏造に注意）。

**3. TDD の健全性** — 実装内部を不自然に写した assert がないか。リファクタ後も意味を保つか。

**4. コメントの正しさ** — コードと一致しているか。誤った機構説明・誤コメントは親が直す。

**5. 自分の手で実行** — 検証コマンド（test/build/typecheck/lint 等）を親が実行し green を確認。既存
テストを壊していないか（独立・全体の両方）。

## 責務境界レビューゲート

返却物 QA を通過した diff は、最終検証に進む前に親が責務境界の軽量チェックを行う。通常 QA に埋もれやすい
設計上の問題をここで分離して扱う。軽い修正では diff を読む中で明らかな責務逸脱がないかだけを確認すればよく、
下記7項目すべてを個別に証明させる必要はない。通常実装以上、または起動条件に該当する差分では
`responsibility-boundary-reviewer` を使う。

### 親が常に見る軽量チェック

- 1つの関数・クラス・モジュールに複数の変更理由が混ざっていないか。
- 入力検証、業務判断、永続化、外部 I/O、表示・レスポンス整形が同じ場所に詰め込まれていないか。
- DB、API、HTTP、ファイル、フレームワークなどの具体実装を上位層が知りすぎていないか。
- 副作用が分散し、再実行・テスト・失敗時の扱いが難しくなっていないか。
- boolean flag や mode 引数、巨大な条件分岐で責務を切り替えていないか。
- 既存の責務配置・命名・ディレクトリ構成から不自然に外れていないか。
- 分割や抽象化が、責務分離ではなく過剰抽象化になっていないか。

### 専用 reviewer の起動条件

次のいずれかに該当する場合は `responsibility-boundary-reviewer` に実装済み diff を渡す。

- 複数層にまたがる変更。
- DB、API、HTTP、ファイル、メール、現在時刻、乱数などの外部 I/O や永続化の追加。
- boolean flag や mode 引数で大きく処理を切り替える変更。
- 新しい抽象化、層、adapter、service、manager、handler、processor などの追加。
- 1つの差分に入力整理、業務判断、副作用、出力整形が混在している疑いがある変更。
- 親 QA で責務混在・境界違反・副作用分散が疑わしいが、通常 QA の観点だけでは判断しにくい変更。

### 判定の扱い

- `問題なし`: ゲート通過。親は通常の最終検証へ進める。
- `軽微`: 親が残課題として明示するか、表層修正で収まる場合だけ親が直す。
- `修正推奨`: `refactor-patch-agent` に reviewer の指摘範囲だけを渡し、最小修正させる。
- `修正必須`: `refactor-patch-agent` に reviewer の指摘範囲だけを渡し、解消するまで最終完了にしない。

### agent の責務分離

`responsibility-boundary-reviewer` は修正しない。判定、問題箇所、理由、影響範囲、最小修正方針、
やりすぎな抽象化を避ける注意点だけを返す。

`refactor-patch-agent` は reviewer の指摘範囲だけを最小修正する。仕様変更、ついで修正、大規模再設計、
新規依存追加、通常実装の代行はしない。修正後は実行した検証と残課題を返す。

## 受け入れ・差し戻しの判断

**親がその場で直してよいのは判断を伴わない表層だけ** — import 整理・命名・コメント・フォーマット。
ここは往復より親が直す方が速く、品質責任は親にある原則とも合う。

**次は hard reject（直さず差し戻す）。** 親がこっそり直すと穴が記録されず再発する：
境界値・異常系の欠落／期待値の根拠不明・捏造の疑い／禁止ファイル変更・既存テスト弱体化・未承認依存／
依頼にない仕様拡張・スコープ逸脱／テスト未実行・失敗テスト放置。
差し戻すときは「何が・なぜ不可で・どう直すか」を AC/観点に紐づけて具体的に書く（「もっと網羅して」は
同じ欠陥を生む）。

## 補足

言語・フレームワーク非依存。プロジェクト固有の検証コマンド（test/build/typecheck/lint 等）や検証
ルールがある場合は、委譲プロンプトの「品質基準」と QA の「自分の手で実行」にその具体を差し込んで使う。
